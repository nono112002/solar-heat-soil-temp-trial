# Pico W + DS18B20 + SDカード + Ambient をつなぎきった日、そしてPCB設計の思考メモ

**2026-04-01 / 太陽熱養生プロジェクト**

---

## やったこと

Raspberry Pi Pico W に DS18B20 温度センサーを7本つなぎ、SDカードに記録しながら Ambient にデータを飛ばす、という一連の流れを動作確認した。
ついでにPCB設計の仕様もほぼ固めた。

結論から言うと、**ファームウェアは動いた。ただし道のりがなかなか険しかった。**

---

## なぜやるか

太陽熱で地面を養生する農業実験で、土中の温度分布を記録したい。
中央・エッジ・深さ方向に合計7本のDS18B20を刺し、30分ごとにSDカードとクラウド（Ambient）の両方に記録する。

電源断でも記録が途切れないこと、現場に行かなくても温度がわかること、この2点がシステムの核。

---

## ハマったポイントと設計の意図

### 1. SDカードがマウントできない問題

最初、SDカードは何度やっても `timeout waiting for v2 card` か `no SD card` で失敗し続けた。

ブレッドボードの配線を疑い、GND接続を確認し、SDカードを変え、ライブラリを入れ直し……それでも失敗。

**原因は2つ重なっていた。**

ひとつは**Pico W の CYW43 WiFi チップが同じ3.3Vレールを共有している**こと。WiFi が動いているとき3.3Vが2.9Vまで落ちていた（テスターで実測）。SDカードの初期化には安定した3.3Vが必要で、これに耐えられなかった。

もうひとつは、もっと単純な話で**拡張ボードの端子台のネジが緩んでいた**。これで接触抵抗が上がり、さらに電圧が落ちていた。

**解決策はシーケンス制御。** `wifi_off()` を呼んでからSDをマウントする。送信のときだけ `wifi_on()` → 送信 → `wifi_off()` という流れにした。WiFiとSDカードが同時に電力を奪い合わない構成にすることで、安定した。

```
WiFi OFF → SD マウント → センサー読み取り → SD書き込み
→ WiFi ON → Ambient送信 → WiFi OFF → 30分待機
```

ソフトウェアで解決したが、PCBでは根本から直す予定（後述）。

---

### 2. センサーIDの管理方法

DS18B20は1-Wireで複数本を同一バスにつなげる。バスをスキャンすると64bitのROMコードが返ってくる。これをどう管理するか。

最初は config.py に直書きしようとしたが、3台のPicoそれぞれでファームウェアを書き換えるたびにIDを確認してコードを書き直す…これはミスが出る。

**SDカードに `sensor_map.json` を置く方法にした。**

```json
{
  "zone": "zone-a",
  "sensors": {
    "286a535200000016": "S1_center_10cm",
    "28983bc0000000c3": "S2_center_25cm",
    ...
  }
}
```

ファームウェアは起動時にSDカードからマッピングを読み込むだけ。Pico3台でファームウェアは共通、SDカードの中身だけ変える。これでヒューマンエラーのリスクが大きく下がった。

---

### 3. ピン配置の設計

GPIO のピン番号を決めるときに気づいたことがある。

Pico W の**物理ピン4〜10が連続している**：

| 物理ピン | GPIO | 用途 |
|---|---|---|
| 4 | GP2 | SD SCK |
| 5 | GP3 | SD MOSI |
| 6 | GP4 | SD MISO |
| 7 | GP5 | SD CS |
| 8 | GND | GND |
| 9 | GP6 | SD カード検出 |
| 10 | GP7 | DS18B20 1-Wire |

SDカードの4本SPI信号＋GND＋カード検出＋センサーバスが**7ピン連続で全部片側に集まる**。

PCBを作るとき、この7ピンをひとつのコネクタでまとめて引き出せる。配線が整理されて、作業ミスも減る。「なんとなくピンを決める」のではなく、**物理的なレイアウトから逆算してピンを決める**のが今回の設計のポイントだった。

## PCB設計の方針（次フェーズ）

動作確認が取れたのでPCBに入る。今回の電源設計は考えることが多かったのでまとめて書いておく。

---

### 電源設計① PicoのWiFiとSDカードを干渉させない

前述のSDカード問題の根本原因はここにある。

Pico W には3.3Vの出力ピン（3V3_OUT）があり、「センサー電源はここから取れる」と思いがちだ。実際そうしていた。**でもこれは罠だった。**

Pico W 内部では WiFi チップ（CYW43）がこの3.3Vレールを共有している。WiFi の送受信時に大きな電流を消費し、レール電圧が瞬間的に落ちる。SDカードもこのレールに繋いでいると、SDの初期化やアクセス中に電圧不足で止まる。

PCBでの根本解決は**3.3Vレールを分離すること**。

```
5V 入力
 ├─ Pico W の VSYS（5V直入れ）→ Pico内部LDOで3.3V生成（WiFi・MCU用）
 └─ MCP1703-3302（外部LDO）→ 3.3V → SDカード・DS18B20専用
```

WiFiが暴れても外部LDOの3.3Vは揺れない。ソフトウェアのシーケンス制御（wifi_off）は保険として残すが、電源で根本から断つ。

---

### 電源設計② 単一入力から3.3Vを生む：LDO選定

外部LDOで5V→3.3Vを作ることにした。最初に思い浮かぶのはAMS1117-3.3だが、今回は使えない。

理由はバックアップ電源の構成にある（後述）。電池バックアップ時、入力電圧が4.2Vまで落ちる。AMS1117はドロップアウト電圧が1.5Vなので、最低4.8Vないと3.3Vが出せない。4.2Vでは力不足だ。

**MCP1703-3302を選んだ。** ドロップアウト250mV。4.2V入力でも 4.2 - 0.25 = 3.95V > 3.3V、十分な余裕がある。

部品一個の選定にも理由がある。それを記録しておくのがこのプロジェクトの趣旨だ。

---

### 電源設計③ バックアップ電源：ダイオードORで瞬時切り替え

フィールドに設置するシステムなので、電源が落ちたとき記録が止まるのは困る。けが人が出るわけではないが、データの連続性は確保したい。

最初はリレーで切り替えることを考えた。でもリレーには**切り替え時間（5〜10ms）と接点の摩耗**がある。なんとなく気持ち悪い。

**ショットキーダイオードOR回路にした。**

```
主電源 5V ─────────────────────────────────┐
                                            ├──→ 5V バス → 各Pico へ
電池(単3×3本=4.5V)─[Schottky: 1N5819]────┘
```

ダイオードは「電圧の高い方から電流が流れる、低い方は逆バイアスで遮断される」という性質を持つ。主電源5V > 電池4.5Vなので、主電源が生きている間は電池側のダイオードが逆バイアスになり電流は流れない。主電源が落ちた瞬間、ダイオードが順バイアスに切り替わり電池から供給が始まる。**機械的な動作なし、遅延なし。**

この構成は上流電源ボックスに1セット置き、3台のPicoBox全体をまとめてバックアップする。

**電池は単3×3本（4.5V）** を選んだ。入手性が高く、交換が簡単。バックアップ持続時間は平均電流30mA（3台合計）として約80時間、3〜4日分。現場に毎日行く必要はない。

---

### 電源設計④ サージ対策

屋外設置なので、雷や電源ON/OFFのスパイクに対する保護が必要だ。

**TVSダイオード（P6KE6.8A等）を5V入力ラインとGND間に入れる。** TVSは過電圧が来たとき瞬時にクランプして後段の回路を保護する。ポリスイッチ（自動復帰ヒューズ）と組み合わせることで、過電流・過電圧の両方に対処できる。

DS18B20の信号線（1-Wire）にも100Ωの直列抵抗を入れる。静電気や誤配線でGPIOピンを壊さないための保護だ。実験環境でケーブルを抜き差しする機会が多いので、ここは省略しない。

---

### 電源設計⑤ 主電源断の検知

バックアップ電源で動き続けながら、「今バックアップ中だ」という情報を送れると便利だ。Picoがメンテナンスが必要なことを自分で知らせてくれる。

GPIOに分圧回路（10kΩ × 2）で主電源5Vを2.5Vに落として監視する。主電源が落ちると GPIO が LOW になり、ファームウェアがそれを検知してAmbientかMQTTにアラートを飛ばす仕組みだ。電池が残っているうちに通知が飛ぶ。

---

### 全体の電源構成まとめ

```
【上流電源ボックス（1台）】
  24V入力
  → TVSダイオード（サージ保護）
  → DC-DC 5V変換
  → 逆接防止ダイオード D1
  → 5V バス ─────────────────────────→ PicoBox A/B/C

  電池(単3×3本=4.5V)
  → Schottky D2（ORダイオード）
  → 5V バス（主電源断時のみ通電）

【PicoBox（3台 同一設計）】
  5V IN
  → ポリスイッチ（過電流保護）
  → VSYS（Pico W） ─ Pico内部LDO ─ WiFi/MCU用 3.3V
  → MCP1703-3302  ─ 外部 3.3V ─ SDカード・DS18B20・LED
  → GP_MON ← 分圧回路 ← 5V バス（主電源監視）
```

シンプルに見えて、各ブロックに理由がある。

## 結果・気づき

- ファームウェアは完成、30秒ごとにSD書き込みとAmbient送信が安定動作
- 電圧問題はソフトウェアで一時回避、PCBで根本解決予定
- 「ブレッドボードのネジが緩んでいた」という物理的な原因が意外と大きかった。**電圧をテスターで実測する習慣は大事**
- ピン番号は連続性を意識して選ぶと配線がきれいになる
